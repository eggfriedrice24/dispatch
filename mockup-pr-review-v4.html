We've been seeing duplicate webhook processing in production when GitHub sends retry events. This PR wraps the handler in a distributed lock using Redis SET NX EX to ensure only one instance processes each event.

Also adds rate limiting to the /api/webhooks endpoint as a defense-in-depth measure.

  • Redis-based lock acquisition with 30s TTL
  • try/finally pattern ensures lock release
  • New rate limiter module for API endpoints
  • Removed legacy handler (superseded)
